## 📝 报告格式规范

### 去重机制

**内容指纹生成:**
- 提取标题 + 核心关键词 (前 50 字)
- 计算 MD5 哈希值作为唯一标识
- 比对历史推送记录

**去重规则:**
- 检查最近 100 条已推送资讯
- 指纹匹配度 > 90% 视为重复
- 重复则自动取热度下一条
- 记录排除原因 (可选)

**已推送记录:**
- 文件：`config/pushed-items.json`
- 格式：`[{fingerprint, title, category, pushed_at}, ...]`
- 保留：最近 500 条记录
- 清理：每次推送后自动清理过期记录

### 版本化规范

**文件命名:**
- Markdown: `daily-top10-YYYY-MM-DD-HHMM.md`
- HTML: `daily-top10-YYYY-MM-DD-HHMM.html`
- 示例：`daily-top10-2026-03-15-1800.md`

**Pages 目录结构:**
```
reports/
├── 2026/
│   ├── 03/
│   │   ├── 15/
│   │   │   ├── daily-top10-2026-03-15-0930.md
│   │   │   ├── daily-top10-2026-03-15-0930.html
│   │   │   ├── daily-top10-2026-03-15-1200.md
│   │   │   ├── daily-top10-2026-03-15-1200.html
│   │   │   ├── daily-top10-2026-03-15-1800.md
│   │   │   └── daily-top10-2026-03-15-1800.html
```

**永久保存原则:**
- ❌ 不覆盖任何历史报告
- ❌ 不删除任何历史版本
- ✅ 每次推送生成新文件
- ✅ 所有版本永久可访问

## 🛠️ 执行流程

```
1. 检查平台状态 (Cookie、可访问性)
   ↓
2. 加载已推送记录 (去重比对)
   ↓
3. 并行抓取各平台数据
   ↓
4. 数据清洗 + 热度计算 + 内容指纹生成
   ↓
5. 去重过滤 (比对历史指纹，重复则取下一条)
   ↓
6. 按领域筛选 (50/30/10/10)
   ↓
7. 提炼要点 + 撰写锐评
   ↓
8. 生成 Markdown 报告 (带时间戳版本)
   ↓
9. 生成 HTML (美观版，带时间戳版本)
   ↓
10. 部署 GitHub Pages (保留历史版本)
    ↓
11. 发送钉钉卡片消息
    ↓
12. 更新已推送记录
    ↓
13. 记录执行日志
```

## ⚠️ 常见错误与反思 (2026-03-15)

### 错误 1：发送模拟数据而非真实抓取
**症状：** 助手编造报告内容，没有真正执行抓取
**原因：** 误以为 Brave API 缺失就无法获取数据，实际上 skill 使用 browser 工具 + X 平台 Cookie
**避免方法：** 始终调用 hot-news-automation skill 执行真实抓取，不手动编造数据

### 错误 2：格式缺少提炼和锐评
**症状：** 报告只有标题列表，没有"要点 + 锐评"
**原因：** 未先读取 report-specs.md 和 SKILL.md
**避免方法：** 每次推送前检查质量清单，确保每条内容有提炼和锐评

### 错误 3：未验证定时任务真实状态
**症状：** 错误报告"任务未执行"，实际已执行
**原因：** 仅查看 `openclaw cron list` 摘要，未深入检查 `jobs.json`
**避免方法：** 诊断时直接读取 `~/.openclaw/cron/jobs.json` 查看完整执行历史

### 错误 4：sessionKey 配置错误
**症状：** 消息投递到私聊而非群聊
**原因：** sessionKey 是 direct 私聊会话，与 delivery.to 群 ID 不匹配
**避免方法：** 群聊投递必须使用 `agent:main:dingtalk:group:cidxxx` 格式的 sessionKey

## 📞 异常处理

### Cookie 过期处理
1. 报告中红色警告
2. 推送提醒消息
3. 等待用户重新登录
4. 恢复后继续推送

### 网络异常处理
1. 重试 3 次 (间隔 30 秒)
2. 降级数据源 (X→Google)
3. 报告中标注"部分数据缺失"
4. 记录异常日志

### GitHub Pages 部署失败
1. 检查部署日志
2. 重试部署
3. 报告中提供 GitHub 源码链接
4. 通知用户
